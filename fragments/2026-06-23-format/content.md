# Fragment Format

> The Sphere fragment is the atomic unit of public knowledge: a bounded, agent-readable fragment with content, provenance, licensing, access policy, payment metadata, and relations to other fragments.

---

## Structure

```
{fragment_id}/
  sphere.json      ← manifest: identity, provenance, access policy, license
  content.md       ← canonical agent-readable Markdown surface
  sources/         ← optional original or intermediate source files
    source.docx
    source.pdf
    export.notion.md
  data/            ← optional structured data sources
    table.csv
    query.sql
    workbook.xlsx
  media/           ← optional media assets
    cover.jpg
    audio.mp3
```

`content.md` is the canonical text surface exposed to agents. Original files may be preserved in `sources/`, `data/`, and `media/` when they are needed for provenance, auditability, citation, or later reprocessing.

Everything else — summary, tags, entities, particles, claims, embeddings, and previews — can be generated by Sphere during processing.

---

## sphere.json — Manifest

The fragment manifest (`sphere.json`) is defined canonically in the `sphere-node` repository, not here. See [`spec/fragment.schema.json`](https://github.com/marianoviola/sphere-node/blob/main/spec/fragment.schema.json) for the authoritative schema and [`spec/node-api.md`](https://github.com/marianoviola/sphere-node/blob/main/spec/node-api.md) for the HTTP contract. This document does not restate the schema; the fields described in prose below are guidance, and the canonical schema governs.

The earlier draft schema in this document used a richer shape (nested `identity`, `hierarchy`, `content_flags`, and `license` objects). The shipped v1 contract is flatter. When they differ, the `sphere-node` schema wins.

---

## content.md — Conventions

- Pure Markdown, no HTML
- Headings start at `##` (H1 is the title, declared in the manifest)
- Images: `![alt](media/filename.ext)`
- Code blocks with language tag
- No inline styling

`content.md` should preserve the meaning of the source material. It may normalize headings, remove layout artifacts, convert tables to Markdown, add source labels, and include transcripts or descriptions. It must not invent claims, citations, authors, rights, or dates.

## Fragment, Source, Particle

Sphere uses three levels:

| Level | Meaning |
|---|---|
| Sphere | The public information space and protocol |
| Fragment | A bounded, publishable unit of public knowledge |
| Particle | A generated internal unit inside a fragment: claim, citation, definition, evidence, table row, media annotation |

Fragments are published and accessed. Particles are generated later for indexing, retrieval, evidence graphs, and quality scoring.

The term "fragment" is intentional: in the Benjaminian sense, knowledge emerges through montage, relation, citation, and constellation rather than through isolated totality. A fragment is not incomplete because it is weak; it is useful because it can enter into meaningful relations.

---

## Informational Hierarchy

Sphere separates administrative, editorial, and semantic structures:

| Concept | Purpose |
|---|---|
| Corpus | Publisher-owned administrative body of fragments |
| Series | Editorial sequence, usually ordered by time or production |
| Constellation | Curated semantic arrangement of fragments around a question, theme, or problem |
| Fragment | Atomic publishable unit |
| Relation | Typed edge between fragments or between a fragment and an external source |
| Particle | Generated internal semantic unit |

`series` answers "what came next?".
`corpus` answers "who owns or administers this body of work?".
`constellation` answers "which fragments make meaning together?".
`relations` answer "how does this fragment connect to another?".

---

## Relations

Relations are explicit semantic links. They are how fragments become a public knowledge space instead of isolated files.

Recommended initial relation types:

| Type | Meaning |
|---|---|
| `cites` | Uses another fragment or source as citation |
| `extends` | Develops or elaborates a prior fragment |
| `responds_to` | Replies to another fragment, text, or claim |
| `updates` | Replaces or revises an earlier fragment |
| `contradicts` | Challenges a fragment or claim |
| `supports` | Provides evidence for a fragment or claim |
| `part_of` | Belongs to a constellation or larger argument |
| `derived_from` | Produced from an original source |
| `translation_of` | Translation of another fragment |
| `same_as` | Canonical equivalence |

Relations can target either a local `fragment_id` or an external URL.

## Source Transformation Rules

### Text Sources

Textual inputs such as `.md`, `.txt`, `.rtf`, `.docx`, Notion exports, PDFs, and copied web pages are converted into the canonical `content.md` when they form one coherent fragment.

Use one fragment when multiple files form one bounded act of knowledge, for example:

- a document plus notes;
- a post plus footnotes;
- a PDF plus extracted references;
- a Notion page plus exported child blocks.

Use multiple fragments when sources are independently citable or should have separate policy, license, attribution, access price, or relations.

Original files may be preserved in `sources/` and declared in `sphere.json`.

### Data Sources

Data files such as `.csv`, `.tsv`, `.xlsx`, `.sql`, `.json`, and database exports are not simply flattened into prose.

Recommended handling:

- Preserve the original file in `data/`.
- Generate a Markdown data note in `content.md` with dataset purpose, schema, columns, units, row count, query context, caveats, and license.
- Include small representative tables only when useful.
- Keep large tables out of `content.md`; expose them as declared data sources.
- For SQL, preserve the query in `data/` and describe database dialect, expected inputs, output schema, and execution assumptions.

If the dataset itself is the product, set `identity.type` to `dataset`.

### Media Sources

Images, audio, and video can be part of a fragment even when they are not linked from prose, but they must not be invisible.

Recommended handling:

- Preserve files in `media/`.
- Declare each media file in `sources` with `kind: image | audio | video`.
- Add alt text, captions, transcripts, shot descriptions, or summaries to `content.md` when possible.
- If media is standalone, include a `## Media` section in `content.md` listing each file and its description.
- Audio/video should include a transcript or description before being used for retrieval-heavy access.

Media with no textual description is valid as preserved evidence, but weak for agent retrieval. The validator may warn about media not referenced in `content.md` or `sources`.

## Corpus and Series Manifests

A publisher can organize fragments into a hierarchy:

```
publisher-root/
  sphere-corpus.json
  series-martech/
    sphere-series.json
    2026-03-27-futuro-martech/
      sphere.json
      content.md
```

### sphere-corpus.json

```json
{
  "sphere_version": "0.4.0",
  "corpus_id": "string",
  "publisher": {
    "name": "string",
    "url": "string | null",
    "sphere_publisher_id": "string"
  },
  "default_license": {
    "type": "CC-BY-NC",
    "llm_retrieval": true,
    "llm_training": false,
    "llm_citation_required": true
  },
  "default_access": {
    "policy": "free"
  }
}
```

Fragment manifests inherit corpus defaults and override only what differs.

### sphere-series.json

```json
{
  "sphere_version": "0.4.0",
  "series_id": "string",
  "title": "string",
  "description": "string | null",
  "language": "ISO 639-1",
  "access_override": null
}
```

## Access Policy Decision Guide

| Scenario | Policy |
|---|---|
| Blog post, no monetization intent | `free` |
| Content with brand/sponsor | `sponsored` |
| Premium content | `paid` |
| Preview free, full paid | `metered` |
| Free, but charge automated access | `metered` with low `price_per_access` |

---

## License Decision Guide

| Intent | License |
|---|---|
| Open, attribution required | `CC-BY` |
| Open, no commercial use | `CC-BY-NC` |
| Open, no derivatives | `CC-BY-NC-ND` |
| Retrieval yes, training no | `proprietary` + `llm_training: false` |
| Full proprietary | `proprietary` + both false |

Default: `CC-BY-NC`, `llm_retrieval: true`, `llm_training: false`.

---

## Fragment Atomicity

A fragment is an autonomous publishable unit. It can be published, updated by replacing its files, or removed entirely. It should not be merged into a parent fragment merely for convenience.

The corpus → series → fragment hierarchy is navigation and policy inheritance. Semantic containment belongs to constellations and relations.
